# Data Dictionary

M1 audit date: 2026-06-09.

This document describes the local KSAS raw dataset used by this project. The
canonical sample inventory is `data/manifests/samples.csv`; the audit summary is
`results/audit/ksas_audit_summary.json`.

## Source And Policy

- Dataset: KSAS-Dataset.
- Source: course-provided local dataset.
- Local raw path: `data/raw/KSAS-Dataset/`.
- Raw-data policy: raw CSV files are local-only and ignored by Git.

The raw CSVs are treated as immutable source files. Any later corrections or
derived arrays must be written outside `data/raw/`.

## Sample Definition

One sample is one segmented recording for:

```text
one participant x one arm x one movement label
```

The audited dataset contains 240 valid samples:

- 20 participants.
- 2 arms.
- 6 labels: five karate block movements plus no-movement/static noise.
- 40 samples per label.
- 120 samples per arm.

## File Naming

Movement files live under `data/raw/KSAS-Dataset/movements/` and follow:

```text
{movement_label_id}-{source_participant_id}-{arm_code}.csv
```

Example: `3-12-i.csv` means movement label `3`, source participant `12`, left
arm.

Committed manifests use pseudonymous participant IDs `P01` through `P20`.
`split_group` is the pseudonymous participant ID, so future train/test splits can
avoid participant leakage.

## Labels

| ID | Label |
|---:|---|
| 0 | no movement |
| 1 | upward block |
| 2 | hammering inward block |
| 3 | extended outward block |

| 5 | rear elbow block |

Each participant and arm combination has all six labels.

## Arms

| Code | Meaning |
|---|---|
| `i` | left arm |
| `d` | right arm |

The dataset README defines the arm used for the execution. The companion KSAS
application instructs users to attach the phone to the forearm in a
runner-style band. Its reference image shows the screen facing outward, the
long device axis approximately aligned with the forearm, and the top of the
phone toward the hand. This is the intended protocol orientation; exact
placement repeatability for every archived recording cannot be verified.

## Channels And Units

Each CSV has 18 numeric Android sensor channels in this exact order.

| Sensor family | Channels | Unit |
|---|---|---|
| Accelerometer | `accelerometer_x`, `accelerometer_y`, `accelerometer_z` | m/s^2 |
| Gravity | `gravity_x`, `gravity_y`, `gravity_z` | m/s^2 |
| Gyroscope | `gyros_x`, `gyros_y`, `gyros_z` | rad/s |
| Linear acceleration | `lin_accel_x`, `lin_accel_y`, `lin_accel_z` | m/s^2 |
| Game rotation vector | `game_rot_vec_x`, `game_rot_vec_y`, `game_rot_vec_z` | unitless |
| Magnetic field | `magn_field_x`, `magn_field_y`, `magn_field_z` | microtesla |

The local dataset README identifies the recording device as Xiaomi Mi A2.

## Sequences

- CSV row order is treated as source sample order.
- The files appear to be already segmented by movement instance.
- No timestamp column is present.
- The acquisition app requests Android `SENSOR_DELAY_GAME`, corresponding to a
  nominal 20 ms interval or 50 Hz. Android treats this as a hint, and no event
  timestamps were retained, so actual rate and jitter are unknown.
- Sequence boundaries are sample indices `0` through `original_length - 1`.
- M1 does not resample, pad, truncate, smooth, or normalize data.
- `processed_length` equals `original_length` in `samples.csv`.

Observed source sequence lengths:

| Scope | Min | Median | Max |
|---|---:|---:|---:|
| All samples | 18 | 33 | 56 |
| Label 0 | 18 | 34 | 52 |
| Label 1 | 24 | 36 | 54 |
| Label 2 | 20 | 29 | 51 |
| Label 3 | 18 | 30 | 48 |
| Label 4 | 25 | 36 | 56 |
| Label 5 | 22 | 34 | 50 |

## Manifest Fields

`data/manifests/samples.csv` is generated by `hmc audit`, not edited by hand.
Important fields:

- `sample_id`: stable project identifier, such as `ksas_m3_p12_i`.
- `filename` and `source_path`: original CSV name and local ignored raw path.
- `checksum_sha256`: raw file checksum.
- `movement_label_id` and `movement_label`: primary target candidate.
- `participant_id`: pseudonymous participant group.
- `arm_code` and `arm`: source arm metadata.
- `recording_id`: source filename stem.
- `split_group`: participant ID for leakage-safe splitting.
- `sensor_location`: `smartphone`.
- `device_model`: `XiaoMi Mi A2`.
- `device_orientation`: empty in the M1 manifest because the manifest predates
  the protocol-source audit; use the documented forearm/device-frame protocol.
- `sampling_rate_hz`: empty because 50 Hz is nominal rather than a measured
  per-recording rate.
- `original_length` and `processed_length`: source and current sequence length.
- `channel_names`: JSON list of ordered channels.
- `quality_flag`: `valid` for every M1-audited sample.

## Exclusions And Quality Flags

No samples are excluded in M1.

The audit found:

- 240 of 240 expected CSV files.
- Expected labels, participants, arms, and participant-arm-label coverage.
- Expected 18-channel schema in every file.
- Numeric values in every channel.
- 0 missing values.
- 0 duplicate rows.
- 0 exact duplicate file checksums.

## Known Measurement Limits

- The nominal requested rate is 50 Hz, but CSVs have no timestamps from which
  to verify realized sampling rate or jitter.
- Six Android sensor streams were registered independently and exported by
  sample index after truncation to a common length; exact cross-sensor temporal
  alignment cannot be reconstructed.
- The app defines intended forearm placement, but placement consistency across
  archived participants cannot be measured retrospectively.
- Preprocessing before these CSVs, if any, is not fully specified by the dataset
  README.
- XROCKET temporal results must report sample spans first. Seconds may be shown
  as approximate values based on nominal 50 Hz.
